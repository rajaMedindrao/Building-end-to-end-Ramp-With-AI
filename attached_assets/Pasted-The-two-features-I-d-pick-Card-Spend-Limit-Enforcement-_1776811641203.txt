## Database Schema

Create these tables in SQLite if they do not already exist. Use this exact schema — do not modify column names.

```sql
CREATE TABLE IF NOT EXISTS employees (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    name TEXT NOT NULL,
    email TEXT NOT NULL UNIQUE,
    department TEXT NOT NULL,
    role TEXT NOT NULL DEFAULT 'employee',
    -- role is either 'employee' or 'manager'
    manager_id INTEGER REFERENCES employees(id)
);

CREATE TABLE IF NOT EXISTS cards (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    employee_id INTEGER NOT NULL REFERENCES employees(id),
    card_name TEXT NOT NULL,
    card_type TEXT NOT NULL DEFAULT 'physical',
    -- card_type is either 'physical' or 'virtual'
    spend_limit_cents INTEGER NOT NULL,
    -- ALL monetary values stored as integers in cents
    -- Example: $1,000.00 is stored as 100000
    status TEXT NOT NULL DEFAULT 'active',
    -- status is either 'active' or 'frozen'
    blocked_categories TEXT DEFAULT '[]',
    -- JSON array of blocked merchant categories
    -- Example: '["gambling", "entertainment"]'
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE IF NOT EXISTS transactions (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    card_id INTEGER NOT NULL REFERENCES cards(id),
    employee_id INTEGER NOT NULL REFERENCES employees(id),
    merchant_name TEXT NOT NULL,
    merchant_category TEXT NOT NULL,
    amount_cents INTEGER NOT NULL,
    -- ALL monetary values stored as integers in cents
    status TEXT NOT NULL DEFAULT 'pending',
    -- status is one of:
    -- 'approved' - passed all checks, charge completed
    -- 'declined' - failed a spend limit or policy check
    -- 'pending_approval' - amount over $500, waiting for manager
    decline_reason TEXT,
    -- populated only when status is 'declined'
    -- Example: 'Exceeds monthly spend limit'
    approval_id INTEGER REFERENCES approvals(id),
    -- populated only when status is 'pending_approval'
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE IF NOT EXISTS approvals (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    transaction_id INTEGER REFERENCES transactions(id),
    manager_id INTEGER NOT NULL REFERENCES employees(id),
    status TEXT NOT NULL DEFAULT 'pending',
    -- status is one of:
    -- 'pending' - waiting for manager action
    -- 'approved' - manager approved
    -- 'rejected' - manager rejected
    requested_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    decided_at TIMESTAMP,
    manager_note TEXT
    -- optional note from manager on approval or rejection
);

CREATE TABLE IF NOT EXISTS monthly_spend (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    card_id INTEGER NOT NULL REFERENCES cards(id),
    month INTEGER NOT NULL,
    -- stored as YYYYMM integer, Example: January 2025 = 202501
    year INTEGER NOT NULL,
    total_spent_cents INTEGER NOT NULL DEFAULT 0,
    -- running total of all APPROVED transactions this month
    -- in cents. Updated every time a transaction is approved.
    UNIQUE(card_id, month, year)
);
```

## Feature 1: Card Spend Limit Enforcement

### What this feature does

Every time a transaction is submitted, the system runs it through a rules engine before approving it. The rules engine checks five conditions in strict order. If any condition fails, the transaction is immediately declined with a specific reason. If all conditions pass, the transaction is approved and the monthly spend total is updated.

This is the core of how Ramp protects companies from overspending.

### The rules engine — strict order of evaluation

Implement these checks in this exact order inside a function called `evaluate_transaction`. Do not change the order. Each rule must be checked independently — do not combine them.

```
Rule 1: Card must be active
    IF card.status == 'frozen'
    THEN decline with reason: "Card is frozen"
    STOP. Do not check remaining rules.

Rule 2: Merchant category must not be blocked
    IF merchant_category is in card.blocked_categories
    THEN decline with reason: "Merchant category is blocked"
    STOP. Do not check remaining rules.

Rule 3: Transaction must not exceed remaining monthly limit
    remaining_limit = card.spend_limit_cents - monthly_spend_this_month
    IF amount_cents > remaining_limit
    THEN decline with reason: "Exceeds monthly spend limit. 
         Remaining: ${remaining / 100:.2f}"
    STOP. Do not check remaining rules.

    IMPORTANT: remaining_limit uses monthly_spend_this_month,
    NOT card.spend_limit_cents directly.
    monthly_spend_this_month comes from the monthly_spend table
    for the current month and year.
    If no row exists for this card this month, treat spent as 0.

Rule 4: Virtual cards have a hard cap of $200.00 per transaction
    IF card.card_type == 'virtual' AND amount_cents > 20000
    THEN decline with reason: "Virtual cards limited to $200 per transaction"
    STOP. Do not check remaining rules.

Rule 5: Transactions over $500.00 require manager approval
    IF amount_cents > 50000
    THEN do NOT decline.
    Instead:
        - Create a transaction record with status 'pending_approval'
        - Create an approval record with status 'pending'
            linked to the manager of the card's employee
        - Return status 'pending_approval' to the frontend
    STOP. Do not mark as approved yet.

If all rules pass:
    - Create transaction record with status 'approved'
    - UPDATE monthly_spend for this card, this month, this year
        by adding amount_cents to total_spent_cents
        Use INSERT OR REPLACE to handle first transaction of month
    - Return status 'approved' to the frontend
```

## Feature 2: Transaction Approval Workflow

### What this feature does

When a transaction is over $500, it enters a pending state and waits for the card employee's manager to approve or reject it. Managers see a queue of pending approvals. They can approve or reject each one with an optional note. When approved, the transaction completes and monthly spend updates. When rejected, the transaction is declined.

This is how Ramp gives finance teams control over large purchases.

### Business logic for approvals

When a manager approves a transaction:
1. Update `approvals` table: set status to 'approved', set decided_at to now, save manager_note
2. Update `transactions` table: set status to 'approved'
3. Update `monthly_spend` table: add the transaction amount to total_spent_cents for this card this month
4. Return the updated transaction with new status

When a manager rejects a transaction:
1. Update `approvals` table: set status to 'rejected', set decided_at to now, save manager_note
2. Update `transactions` table: set status to 'declined', set decline_reason to "Rejected by manager: [manager_note]"
3. Do NOT update monthly_spend — rejected transactions do not count toward spend
4. Return the updated transaction with new status

Validate that the manager_id in the request matches the manager_id on the approval record. If not, return HTTP 403 with message "You are not authorized to approve this transaction."

**GET /api/approvals/history**

Query parameter: manager_id (integer)

Returns all approvals (approved and rejected) for this manager, most recent first, limit 20.

## Acceptance criteria

The build is complete when:

1. Submitting a transaction under the monthly limit and under $500 returns "approved" and the card's remaining limit decreases correctly
2. Submitting a transaction that would push the card over its monthly limit returns "declined" with the correct remaining amount in the message
3. Submitting a transaction over $500 returns "pending_approval" and creates a record in the approvals table
4. A frozen card declines all transactions immediately
5. A blocked merchant category declines the transaction immediately
6. A virtual card declines any single transaction over $200
7. A manager can see their pending approval queue filtered to only their employees
8. Approving a transaction updates the transaction status to approved and updates monthly spend
9. Rejecting a transaction updates the transaction status to declined and does not update monthly spend
10. The cards overview table shows correct utilization percentages that update after each transaction
